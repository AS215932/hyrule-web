{# ── 2. Terminology ──────────────────────────────────────────────────────── #}
2. terminology

The key words in this document.

The key words MUST, MUST NOT, and MAY in this document are to be interpreted as described in BCP 14. The operator MUST NOT ask who you are — there is no column in the schema to write your email into. Payment MUST settle before provisioning; nothing is invoiced, everything is prepaid. Failed paid work MUST NOT be charged. Customers MAY be autonomous agents, and increasingly are. Agents MAY be humans. We don't check. That's the point.

{# ── 4. Network Considerations ───────────────────────────────────────────── #}
4. network considerations

We run our own ASN. Not a reseller.

4.1

Our own ASN

AS215932 on RIPE. Prefix 2a0c:b641:b50::/44. Transit via AS34872 + AS210233. Routing policy lives in this same repo.

4.2

IPv6-native + NAT64

Every VM gets a routable /128 inside 2a0c:b641:b51::/48. NAT64 + DNS64 on rtr reaches IPv4 destinations transparently.

4.3

Boring open stack

FRR, Knot, XCP-NG, Caddy, OpenBSD, nftables, WireGuard, Prometheus, Vault. No proprietary control plane. See the inventory →

4.4

Crypto-native, no KYC

USDC on facilitator-verified chains{% if native_methods %}, plus {{ native_methods | join(" and ") }}{% endif %}. Payment rails are fetched from the backend at order time and never hardcoded in the UI.

4.5

x402 for agents

The API ships a `/.well-known/x402.json` manifest. Agents can discover, price, and pay per-call. POST /v1/vm/create with an `X-Payment` header is the whole flow. Read §agents →

4.6

Solo-operated, transparent

One operator in NL, compute in OVH FR, routing across Servperso NL+DE. Every config is in a public-shaped repo. Read the FAQ →
